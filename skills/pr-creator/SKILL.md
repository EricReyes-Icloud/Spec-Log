## Activation Contract

Activate this skill when the user asks you to:
- Write, create, redact, or prepare a PR description
- Generate content for a GitHub pull request
- Document changes in PR format
- "Haz la PR" / "redacta la PR" / "crea el contenido de la PR"

Do NOT activate when the user asks to open, merge, review, or submit a PR (those are execution actions, not content creation).

## Hard Rules

1. **Language**: Write the entire PR in English. PRs are technical artifacts — the Language Domain Contract applies.
2. **Do NOT translate code names**: File paths, variable names, function names, collection names, route paths, parameter names, and any other code identifiers MUST stay in their **original form** as they appear in the codebase. Even if they are in Spanish.
3. **Do NOT include** information that is not relevant to the PR, such as:
   - AI attribution ("Co-Authored-By", "generated by", etc.)
   - Emojis unless the user explicitly requests them
   - Unnecessary markdown flair
4. **File path format**: Use backtick-wrapped relative paths from repo root (e.g. `` `src/app/page.tsx` ``).
5. **Be concise**: Each section should be informative but not bloated. Prefer bullet points under Changes Made.
6. **Anchor in facts**: Every statement in the PR must be verifiable from the diff. Do not speculate about impact you cannot demonstrate.

## Decision Gates

| User says | Action |
|-----------|--------|
| "create/redact/write the PR" | Produce the PR content in the four-section format, save to `openspec/PR-{number}-{slug}.md`, and return the content inline. |
| "pero en español" / "but in Spanish" | Same format, translated to neutral professional Spanish. |
| "solo dime los cambios" / "just list changes" | Skip the full format, provide a compact bullet list. |
| Format unspecified | Use the four-section format (Description, Changes Made, Impact, Notes). |

## Execution Steps

1. **Understand the diff**: Read `git log --oneline -10` and `git diff` (or staged changes) to understand what changed. If the PR involves multiple commits, review all of them, not just the latest.
2. **Categorize changes**: Group changes into logical categories (Bug fixes, Features, Dev tooling, Configuration, Refactors).
3. **Write the four sections**:

### Description
A 2–3 paragraph executive summary. Answer:
- What problem were we solving?
- Why was it a problem?
- What is the high-level approach of the fix?

### Changes Made
Bullet-point list grouped by category. For each change:
- File path in backticks
- What changed (one line)
- Why (optional, if not obvious from Description)

Group by: Bug fixes, Features, Dev tooling, Configuration, etc.

### Impact
Short paragraphs or bullet points covering:
- What behavior changes for users/systems
- Backward compatibility
- Dev experience improvements
- Any risks or caveats

### Notes
Optional but recommended:
- How to test / verify
- Known follow-up issues (link to TODOs or issues)
- Dependencies on other PRs or deployments

4. **Save to `openspec/`**: Save the PR content to `openspec/PR-{number}-{slug}.md`.
   - Number: next sequential PR number (check existing `openspec/PR-*` files).
   - Slug: kebab-case summary of the change.
5. **Return inline**: Show the user the full PR content inline so they can copy-paste to GitHub.

## Output Contract

Return:
- The file path where the PR was saved: `openspec/PR-{number}-{slug}.md`
- The full PR content inline for the user to review and copy.
